// SQL generated automatically: SELECT COUNT(*) > 0 FROM user WHERE email = ?

// For UPDATE (exclude self): existsByEmailAndIdNot(email, id) // WHERE email = ? AND id != ?

JOIN FETCH Explained

Without JOIN FETCH, Spring fires N+1 queries — 1 for locations, then 1 extra per user to load their data.

❌ N+1 Problem: SELECT * FROM location WHERE province_code = ? -- then for EACH location: SELECT * FROM user WHERE id = ? × N times!
✅ JOIN FETCH Solution: @Query("SELECT l FROM Location l JOIN FETCH l.user WHERE l.provinceCode = :code") -- ONE SQL query fetches everything!

1. Explain the ERD and its relationships
We have 6 entities: Department, Course, User, Location, Submission, and PlagiarismCase. The relationships are: One-to-Many between Department and Course (one department has many courses), Many-to-One the reverse. One-to-One between User and Location (each student has one location). Many-to-Many between PlagiarismCase and Submission via the case_submissions join table. Additionally User has One-to-Many with Submission, and Course has One-to-Many with Submission.

2. How does saving a Location work?
The saveLocation() method is annotated with @Transactional, ensuring atomicity. We first fetch the User by ID, then create a Location with @OneToOne @JoinColumn(name="user_id", unique=true) pointing to that user. We also call user.setLocation(location) for bidirectional sync — this keeps the in-memory Java objects consistent even before the next database read. The unique=true on the JoinColumn creates a database-level UNIQUE constraint preventing duplicate locations per user.

3. How does Sorting work in Spring Data JPA?
Sorting uses Sort.by(sortBy).ascending() or .descending(). When passed to findAll(sort), Spring Data JPA automatically adds ORDER BY to the SQL query. I have three methods: getAllSubmissionsSorted() for sorting only, getSubmissionsPaginated() for pagination only using PageRequest.of(page, size), and getSubmissionsWithPaginationAndSorting() that combines both using PageRequest.of(page, size, sort). This generates a single optimized SQL query with LIMIT, OFFSET, and ORDER BY.

4. How does Pagination improve performance?
Without pagination, fetching all 1000+ records wastes memory and network bandwidth. Page from Spring Data JPA returns only the requested slice using SQL LIMIT and OFFSET. It also automatically runs a COUNT(*) query to calculate totalPages and totalElements. The response includes metadata: pageNumber, pageSize, totalElements, totalPages, first, last, hasNext, hasPrevious — so the client knows exactly where it is in the dataset.

5. Explain the Many-to-Many join table
PlagiarismCase owns the relationship and has @ManyToMany with @JoinTable(name="case_submissions", joinColumns=@JoinColumn(name="plagiarism_case_id"), inverseJoinColumns=@JoinColumn(name="submission_id")). This creates a join table with two foreign key columns. Submission has @ManyToMany(mappedBy="submissions") — it's the inverse side and doesn't control the join table. I added helper methods addSubmission() and removeSubmission() to keep both sides of the in-memory relationship synchronized.

6. Explain the One-to-Many foreign key usage
In Department, @OneToMany(mappedBy="department", cascade=CascadeType.ALL, orphanRemoval=true) tells JPA there's a collection of courses. The mappedBy points to the 'department' field in Course. In Course, @ManyToOne @JoinColumn(name="department_id") creates the actual foreign key column in the course table. Cascade ALL means deleting a Department also deletes its Courses. orphanRemoval=true means removing a Course from the list also deletes it from the database.

7. How does the One-to-One relationship work?
Location OWNS the relationship — it has @OneToOne @JoinColumn(name="user_id", unique=true). The unique=true creates a UNIQUE constraint in the database ensuring one user can only have one location. User has @OneToOne(mappedBy="user") which is the inverse side — it just references the relationship. When saving, we set location.setUser(user) AND user.setLocation(location) for bidirectional synchronization so the Java object graph stays consistent.

8. How does existsBy() optimize duplicate checking?
Instead of fetching the full User object (SELECT * FROM user WHERE email=?), existsByEmail() generates SELECT COUNT(*) > 0 FROM user WHERE email=?. This returns a boolean using a COUNT query — much faster as no object needs to be constructed. For UPDATE scenarios, existsByEmailAndIdNot(email, id) adds WHERE email=? AND id!=? to exclude the current user being updated. When a duplicate is found, we throw DuplicateResourceException which is caught by @RestControllerAdvice and returns HTTP 409 Conflict.

9. Explain the Province query and JOIN FETCH
Without JOIN FETCH, JPA uses LAZY loading: one query for locations, then one extra query per location to load the User — this is the N+1 problem. With @Query("SELECT l FROM Location l JOIN FETCH l.user WHERE l.provinceCode = :code"), JPA generates a single SQL JOIN that fetches both Location and User data in one round trip to the database. The LOWER() function makes the name search case-insensitive, and CONCAT('%', :name, '%') enables partial matching. Results are mapped to ProvinceUserResponse DTO with a list of UserSummary objects.
